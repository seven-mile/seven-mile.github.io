Servicing Stack Research Report

10月 14, 2019

综述

何为「服务堆栈」(Servicing Stack，简称为 Stack)？服务(Servicing)是指 Windows 系统中的高级管理、部署行为，针对的对象是「系统映像」(Image，包括在线(Online)映像和离线(Offline)映像)，广义上包括许多安装时的部署步骤、版本管理、Appx程序包管理、驱动包管理和更新包管理。但服务堆栈中的服务是狭义地指代更新包管理(Package Servicing，更新包以下简称包)。

服务堆栈作为 Windows 的一个核心组件，地位是非常重要的，因为 Windows 从 Vista 开始就是全盘组件化(Componentized)的，系统中的任何文件，包括 ntoskrnl.exe 、ApiSet 和服务堆栈自己等等，都可以进行更新。这个组件存储池(Component Store)叫做 WinSxS ，就是那个经常困扰得用户总想把它删除的大块头。

资深 Windows 用户大都知道一个用户，同时也是一个进程，一个服务，TrustedInstaller(以下简称 TI)。本质上这个东西是一个双重意义的壳子。

首先 TI 是服务堆栈的一部分，实际上，是一个 COM RPC API 存根。你可能已经注意到，服务堆栈太强大了，尤其是它可以操作在线系统（当前启动的系统）的组件池，所以它要进行权限隔离。于是它用自己的一套公开接口（虽然是公开的但没有文档）重新封装了一遍服务堆栈的公开对象。在客户端操作服务堆栈的时候，操作离线系统时，可以不使用 TI ，但操作在线系统时，微软是希望客户端使用这一 RPC API 进行跨进程调用的，如此一来你拿不到 TI 权限，但却能调用服务堆栈 API 。事情到这里还没完，TI 自己并不是你调用的对象——你没有足够的权限启动它。它需要从 TI 服务启动才能得到正确的权限，所以微软又做了一个壳外壳——CbsApi.dll，它是一个真正的空壳，作为 RPC 调用的 COM DLL ，把一个 COCLASS 和不少公开接口注册到了注册表里。到这里你可能以为终于完了，但其实还是有一层封装……那就是 TiWorker ，顾名思义，TI 毕竟是个壳，它并不直接与服务堆栈 API 交互，而是靠与 TiWorker 通讯。TiWorker 由 TI 自己启动，继承了 TI 的权限，可以加载服务堆栈的 API DLL 并顺利调用。所以总结一下，一个客户端（Windows 更新、程序与功能、DISM、程序包管理器等等）想要操作在线映像，真实的通讯路径是这样的：

$$
\ce{\text{Client}->T[Load DLL]\text{CbsApi.dll}->T[RPC]\text{TI}->T[RPC]TiWorker->T[Load DLL]\text{Servicing Stack}}
$$

其次它作为用户仅仅出现在文件、注册表等对象的所有权中，而拥有这份权限的进程的用户必然是 System ，并不存在以 TrustedInstaller 为用户的进程，但这并不意味着 System 用户所有的进程可以操作 TI 的东西。TI 权限在进程中的体现是一个叫做 NT SERVICES\Trusted Installer 的 Token 。到这里你应该已经猜到了，正常的 Windows 下只可能有 TI 和 TiWorker 拥有 TI 权限。但有没有其他方法呢？当然有！其中的技术细节偏离主题，不再赘述，我推荐你直接使用 NSudo 项目 ，尤其可以直接用它启动 VS ，这样你编写的客户端（是的，在这篇报告中，我们会撰写一个与服务堆栈交互的客户端）就可以像操作离线系统那样操作在线系统，自己喧宾夺主化身为 TiWorker （雾），这给调试带来的利好是巨大的。

说回组件存储池，实际上它在 XP 时代已经出现，当时开发的口号是「下一代 System32」，虽然到现在也没有取代 System32 ，但它确实成功实现了多版本 DLL 共存，并且可以随时回退，为微软解决了大量的问题，简化了许多流程。最典型的例子就是通用组件库 V5 和 V6 的切换了，众所周知在 XP/Vista/7 盛行的年代，许多应用程序的界面是 Windows 经典的可视化风格，但实际上只需要在资源文件中绑定一个 manifest 指定使用通用组件库 V6 ，就可以加载出当下主题的可视化风格。到了更高版本的 Windows ，还把可选功能(Capability)集成了进去，并且存储池拥有自我修复功能，所以与我们熟知的工具 sfc.exe 也是紧密相连的（sfc.exe 基于 WRP Integrity API，是服务堆栈的一部分），在 MVP 手中是一把威力巨大的利刃。

组件存储池并非我们的主题，但我们还是了解一些它的内部结构为好。它也有公开的 API ，来自 sxs.dll 和 sxsstore.dll ，这个机制也叫 Fusion 。它可以管理各类运行时的不同版本、COM 组件的非注册性调用等等（通过重定向）。它底层为节省空间做出了一些贡献，例如硬链接技术的使用（但你无法在资源管理器中看到硬链接的节省空间数）。它是由程序集(Side by Side Assembly)组织的。我逆向了 ISxSStore 接口，但从未使用过它，它的定义如下：

从包说起

那么什么是包呢？Windows 更新当然是包，除此之外它还包括 Windows 功能(Feature)和可选功能(Capability)，封装格式主要包括 *.cab *.msu ，解包后还有自描述文件 *.mum ，是一个 xml 格式的 manifest 。

每个包有一个唯一标识符(Identity)，格式为 [Product]-[Feature]-Package~[PublicKeyToken]~[Architecture]~~[Version] （功能包）或 [Product]-[Feature]-Package~[PublicKeyToken]~[Architecture]~[Region]~[Version] （语言包）或 Package_for_[UpdateName]~[Architecture]~[Region]~[Version] （更新）。

“Product” 指的是包针对的产品名称，绝大多数时候是 Microsoft-Windows 和 Microsoft-Onecore ，少数时候是 OpenSSH-Client 这类东西。

“PublicKeyToken” 是一个签名公钥令牌，几乎总是为31bf3856ad364e35，微软用它很久了，你应该也在许多 .Net 程序集里看到过它。

“Region” 就是语言区域，类似 en-us zh-cn 这样的字符串，但非语言包一般都为空，所以在格式中会突兀地出现 ~~ 。

“Version” 也是微软长久以来使用的版本书写方式，称为 Four Parts Version 具体格式为 Major.Minor.Build.Revision 。

“UpdateName” 是更新名，常是 KB (Knowledge Base) 文章序号，还有一类是 Rollup Fix ，是微软一直维护的积累性更新包，一般来讲你只需要安装最新的 Rollup Fix ，而更新 Rollup Fix 时旧的包会被取代(Superseded)

每个包还有一个重要的参数是释放类型(Release Type)，包括 OnDemand Pack（可选功能包）、Language Pack（语言包）、Foundation（基础包，实际上是所有 Windows 功能的母包）、Update（积累性更新）、Security Update（安全更新）。

自然而然地，包当然有安装状态。从这里开始我们要讨论一些内部实现相关的东西了，不再脱离实现泛泛而谈。安装状态在服务堆栈里有多种表达方法，用于外部交互的是 _CbsInstallState，而在内部使用的是 CbsState ，它们的部分常量值有差别。

是的，我只是向你介绍这样一个现象，相信你也并未被这两个简单的枚举类型吓到。

这是一种关于包很重要的常量，常见的值有 Installed Absent Staged Superseded ，其它的状态较为少见。Absent 就是并未安装的游离态，Staged 就是存储进存储池了，类似于 Git 中的 Staged 状态，而 Superseded 是旧包被新包取代，这样的两个包一般是 Revision 版本不同。

值得一提的是 Invalid 这类状态，我还没碰上过，也不知到底是怎么个非法。

理解服务堆栈

虽然没有就包在文件系统中的存在形式做深入探讨，但彻底了解了包在堆栈中的常用封装的我们已经可以开始理清服务堆栈的结构了。服务堆栈并不把自己放入 System32 ，而是始终存在于 WinSxS 中，以组件的形式存在，我们的系统中时常有多版本的服务堆栈。一个典型的堆栈文件夹路径是 %WINDIR%\WinSxS\amd64_microsoft-windows-servicingstack_31bf3856ad364e35_10.0.18362.1_none_dbbcd1444482e944 ，经过刚刚包的标识符讨论，你应该可以理解路径中的大部分元素意义。none 指的是 NoneSxS 组件，不过这个我们不深究。

依然是顾名思义，既然叫 堆栈(Stack，可见微软文档的中文翻译者文学素养之高，这可是现代汉语的偏义复词（雾）) 那么就具有栈式结构。栈顶是我们曾提到的客户端，像是 Windows 更新、Windows 语言包设置、甚至是 Win10 的开发者模式。

紧接着往下是 CBS(Component-Based Servicing) ，也即基于组件的服务。它是一个广义的概念，TI 和 TiWorker 实际上属于它的一部分，但只是壳子，并非核心。CBS 的核心 DLL 非常直白，就叫 CbsCore.dll 。刚刚说的 TiWorker 所加载的服务堆栈 DLL 主要就是指它。它实现了服务堆栈的大部分接口，具体的接口会在以后讨论，我们要知道的是与 CBS 交互的核心接口被叫做 Session(会话，对应接口为 ICbsSession)。 CBS 管理的存储池层面叫做 CBS Store ，主要是注册表部分，也就是 COMPONENT Hive。

CBS 使用什么来完成它的工作呢？在服务堆栈的栈底，是 CSI(Component Servicing Infrastructure) 它的核心是 wcp.dll ，WCP(Windows Componentization Platform) Windows 组件化平台，实现了成千上万的、海量的函数、功能与接口，用以支持组件的部署与管理。CSI 自己的存储池层面抽象出来被叫做 CSI Store ，而与 CSI 交互，核心的概念被叫做 Transaction(事务，对应接口为 ICSITransaction)。

一个区分 CBS 与 CSI 工作的关键点在于，CBS 维护的是注册表中的组件存储，而 CSI 维护的是文件系统中的（WinSxS 文件夹）。

CBS 所处理的东西并没有 CSI 那么恐怖，所以还是可以作为必读内容来写的。我们先拿一个典型的、完整的包安装过程来举例说明堆栈的前2.5层如何工作。

首先作为一个称职的客户端，从网络上辛苦地下载好了一个包，就假定是一个 cab ，并且未在系统里安装过。你接下来就会初始化好 CBS DLL 和一个 ICbsSession 接口（这些细节先不管），调用它的 CreatePackage 方法用 cab 文件创建一个 ICbsPackage 对象。这个方法的原型是

主要关注 _CbsPackageType ，对于 cab 包我们可以提供一个 CbsPackageTypeCabinet = 0 给它，而包路径就写文件路径，后一个参数填 NULL，把 ICbsPackage* 用你的变量存起来。你就这样轻松地得到了未安装的一个 cab 包的抽象对象。

注意到这两个函数，你需要首先对包进行评估(Evaluate)，以得到可以应用的状态(Applicability State)和当前的状态(Current State)。按理来讲一个正确的包会分别给你返回 CbsInstallStateInstalled 和 CbsInstallStateAbsent ，然后你调用 InitiateChanges 来初期化更改，installOptions = 0, targetState = CbsInstallStateInstalled，注意，仅仅是初期化，并未进行更改，这就是 Session 设计的精髓，以类似对话的形式存下想要的更改。怎么结束对话呢？结束我们的 ICbsSession 即可。这时，在会话的结束过程中一切更改将会被应用于系统，上面初期化更改时你提供的 ICbsUIHandler* 就会在这时被回调调用，让你得知进度，这个以后再介绍。有时一些更改需要重启，会话此时还会告诉你，你需不需要重启。

这样一来，CBS 的工作也就到此结束，你发现它所做的事情大概有这么几种：加载读取包信息、评估包状态，它把要更改的包扔给 CSI 来完成实际操作，还负责安装时的记录工作，生成安装事件、有时还在“程序与功能”中加入此包。除此之外，作为一个包管理器，它公开了枚举组件存储池中的包的接口。

这就是 CBS 所完成的大多数工作了，当然，到新版本还有腐败侦测与修复功能。

以下为选读内容

CSI 基于 Transaction ，这暗示 CSI 的操作基于 KTM(Kernel Transaction Manager) ，一个公开且有文档的内核机制，不过这已经太过底层而不适于交互。

CSI 在安装组件时有时需要采用高级安装器(Advanced Installer)，这是通过 CMI(Component Management Infrastructure) 来完成的。它基于 COM 有一个 API ，并且类型库是公开的，其 DLL 在 %WINDIR%\AdvancedInstallers\cmiv2.dll 下。我并未尝试过使用它。

而 CSI 关于驱动的安装交给了 DMI(Driver Management and Install) ，来调用 DriverStore。这并不是我们讨论的主题，就此略过。

最后关于注册表的一些零星操作由 SMI(Systems Management Infrastructure) 处理。

这是一些无关紧要的边角料，接下来我们来细细讲一讲 CSI 安装一个组件的过程。

CBS 确定好要更改的组件/包，把它传递给 CSI ，然后 CSI ：

  • 如果是安装一个组件，CSI 首先要 Stage 它，也即存储。在存储时，CSI 检查包的完整性、与已存在的对应文件对比以决定要加入那些文件、理清组件间的依赖关系，然后再真正进行安装，并在最后按以上结果评估出一个正确的包的状态。
  • 如果是卸载一个组件，CSI 首先检查组件中正在被使用的文件，判断是否需要重新启动等，然后把当前组件及其依赖项的文件置为非可用状态。注意，此时这些文件并不一定会被从磁盘上抹除，也有可能接着保留。

在两种更改的过程中，都不仅仅有文件的移动或删除，还有一些 cmd 命令和注册表更改要执行，CSI 把一切需要执行的更改全都维护在一个队列中，叫做 Primitive Operations Queue(POQ) 。如果安装过程需要中断，CSI 会序列化(Serialize)这个队列，把它存在 %WINDIR%\WinSxS\Pending.xml 中。

最后，介绍完了服务堆栈的两大核心 CBS 和 CSI ，我们可以看看堆栈中其它重要的小组件，例如 dpx.dll ，这是唯一一个与包相关的、存在公开头文件（dpx.h, dpx1.h, 附于 Windows SDK 内）的 DLL ，全名是 Delta Package eXpander，它可以展开 Delta 格式包。

最重要的是 wdscore.dll ，也就是 Panther Engine ，黑豹引擎模块，负责了整个堆栈的日志工作和一些实用工具。CBS.log 和 Dism.log 都是由它记录的。

调试服务堆栈，日志和逆向是我们最有用的信息源。
